# Authentication

**Status:** current

Email/password authentication via [better-auth](https://www.better-auth.com),
with per-user data isolation. Anyone can sign up; each account only ever sees
its own products and entries.

## Pieces

- `src/lib/auth.ts` — better-auth server config (Drizzle/Neon adapter,
  email+password, min length 8, Resend-backed `sendResetPassword`).
- `src/app/api/auth/[...all]/route.ts` — the better-auth request handler.
- `src/lib/auth-client.ts` — React client (`signIn`, `signUp`, `signOut`,
  `requestPasswordReset`, `resetPassword`, `changePassword`).
- `src/lib/auth-helpers.ts` — `getSessionUser()` and `requireUserId()` for
  server components and actions.
- `src/middleware.ts` — redirects unauthenticated users to `/login` and
  authenticated users away from the auth pages (optimistic cookie check).
- `src/db/auth-schema.ts` — `user` / `session` / `account` / `verification`
  tables (generated by the better-auth CLI), re-exported from `db/schema.ts`.

## Routes

- `(auth)` group (public): `/login`, `/signup`, `/forgot-password`,
  `/reset-password` — centered layout, no app chrome.
- `(app)` group (protected): dashboard, `/products`, `/entries`, `/account` —
  the layout fetches the session and renders the header + user menu.

## Data isolation

`products.userId` references `user.id` (cascade delete). Entries and expenses
inherit ownership through their product. **Every** server action calls
`requireUserId()` and scopes its queries:

- product reads/writes filter or set `userId`;
- record create/update/delete verify the target product is owned by the user
  (returning a friendly error otherwise);
- dashboard and entries queries inner-join products filtered by `userId`.

Defense in depth: the middleware gates routes, the `(app)` layout re-checks the
session, and the actions re-check ownership — so a forged request still can't
read or mutate another user's data.

## Password flows

- **Forgot:** `/forgot-password` → `requestPasswordReset({ redirectTo:
  "/reset-password" })` → better-auth emails a tokenized link via Resend.
- **Reset:** the link lands on `/reset-password?token=…` → `resetPassword`.
- **Change:** `/account` → `changePassword` (current + new, revokes other
  sessions).

If `RESEND_API_KEY` is unset, the reset link is logged to the server console so
the flow still works in development.

## Environment

See [`.env.example`](../.env.example): `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`,
`RESEND_API_KEY`, `EMAIL_FROM`. Auth tables are created by `pnpm db:push`.
